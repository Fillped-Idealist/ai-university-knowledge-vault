# Codex配置示例

类型:: 配置示例
状态:: 进阶可选
阶段:: 熟悉基础后
关联目标:: [[MCP进阶连接指南]]
下一步行动:: 先读完 MCP 指南，再决定是否配置

## 先说明

这篇不是入门教程。刚下载资料包的人，先看 [[完整操作手册]] 和 [[AI使用指令集]]，用复制指令跑通就够了。

当你已经有一批真实笔记，并且想让 Codex 这类工具读取本地资料包时，再看这里。

## 你需要准备什么

- 一个已经能正常打开的 Obsidian 资料包。
- 一份测试库，里面不要放隐私资料。
- 一个可用的 MCP 服务或 Obsidian MCP 插件。
- Codex 的配置文件位置。

不同电脑、不同客户端的路径可能不一样。不要照抄别人的绝对路径，要换成自己的资料包路径。

## 配置思路

MCP 配置一般包含三件事：

1. 服务名称：你给这个连接起的名字。
2. 启动命令：客户端用什么命令启动 MCP 服务。
3. 参数：通常包含资料包路径、权限、端口或插件设置。

示例结构可能类似下面这样。它只是结构示例，不代表你电脑上一定能直接运行：

```toml
[mcp_servers.obsidian_vault]
command = "your-mcp-command"
args = [
  "--vault",
  "/Users/你的用户名/Documents/ai-university-knowledge-vault-test",
  "--readonly"
]
```

如果你的 MCP 服务要求环境变量，可以写成这种思路：

```toml
[mcp_servers.obsidian_vault]
command = "your-mcp-command"
args = ["--vault", "/你的/测试库/路径"]
env = { OBSIDIAN_API_KEY = "${OBSIDIAN_API_KEY}" }
```

重点是：密钥不要直接写进配置文件，更不要提交到 GitHub。

## 推荐测试问题

配置后先问只读问题：

```text
请读取我的 Obsidian 测试资料包，只做检查，不要修改文件。
请告诉我：
1. 根目录有哪些文件夹。
2. 是否能找到 README.md。
3. 是否能找到 🏠 大学全景图.md。
4. 哪些文件包含 状态:: 规划中。
5. 不要写入任何内容。
```

如果只读测试成功，再做一个小范围写入测试：

```text
请只修改测试库中的 1 个文件：复盘周记.md。
修改内容：
1. 补一个 类型:: 周记。
2. 补一个 状态:: 待复盘。
3. 在文末增加 3 条下周最小行动。
修改前请先列出计划，等我确认后再动手。
```

## 排查顺序

出现问题时，按这个顺序查：

1. Obsidian 能不能正常打开资料包。
2. 配置里的路径是不是资料包根目录。
3. MCP 服务是否能独立启动。
4. Codex 是否读到了配置文件。
5. 是否因为权限限制只能读不能写。
6. 是否把密钥写错或没有设置环境变量。

不要一出错就重装所有东西。先把错误信息复制出来，让常用智能助手逐条解释，通常能很快定位。

## 可以让智能助手帮你做什么

```text
我正在给 Codex 配置 Obsidian MCP。
请你先不要让我直接运行危险命令。
请根据我提供的信息帮我：
1. 判断配置文件应该放在哪里。
2. 检查 vault 路径是否正确。
3. 把配置改成只读优先。
4. 标出哪些内容不能提交到 GitHub。
5. 给我一个最小测试问题。
```

## 安全提醒

- 第一次只连接测试库。
- 第一次只读，不写入。
- 第一次写入只改 1 个文件。
- 不把 Token、API Key、密码、Cookie 放进仓库。
- 批量修改前先备份整个资料包。
