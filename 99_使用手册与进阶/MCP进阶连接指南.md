## MCP 能解决什么

MCP（模型上下文协议，可以理解为让智能助手连接本地工具和文件的接口）解决的是“反复复制粘贴资料”的问题。

没有 MCP 时，你通常要这样做：打开笔记、复制内容、粘贴给工具、等它整理、再手动放回 Obsidian。知识库小的时候没问题，笔记多了就很烦。

配置 MCP 后，在你授权的范围内，常用工具可以直接读取本地资料包，帮你做这些事：

- 搜索和某个目标有关的多篇笔记。
- 检查哪些文件缺字段、缺链接、缺下一步行动。
- 根据周记更新长期笔记。
- 把实习、竞赛、科研经历整理成简历素材。
- 批量找出孤立笔记和重复文件。

它不是入门必需品。前 2-4 周先用复制指令就够了，等资料包里有几十篇真实笔记后再配置更合适。

## 配置前先判断自己需不需要

适合配置：

- 已经稳定记录 2-4 周。
- 知识库里有 30 篇以上真实笔记。
- 经常需要跨文件整理，比如求职、保研、考研复试前集中梳理材料。
- 能分清公开资料和私人资料。

暂时不适合配置：

- 刚下载资料包，还没写自己的内容。
- 只是想试试新功能。
- 不确定哪些文件包含隐私。
- 没有备份习惯。

## 安全设置原则

1. 先复制一份测试库，不直接连接私人库。
2. 先只读，确认能搜索文件后，再考虑写入。
3. 不把 API Key、Token、密码、证件号、完整成绩单写进公开文件。
4. 批量修改前先让工具列计划，再确认执行。
5. 修改后用 Git 或压缩包备份，能退回上一版。

## 推荐配置流程

### 第一步：准备测试库

把这个资料包复制一份，命名为：

```text
ai-university-knowledge-vault-test
```

测试库里只放模板和脱敏案例。真实成绩、学校、姓名、聊天记录、申请材料先不要放进去。

### 第二步：确认资料包路径

在电脑里找到测试库文件夹路径。比如 Mac 上可能类似：

```text
/Users/你的用户名/Documents/ai-university-knowledge-vault-test
```

Windows 上可能类似：

```text
C:\Users\你的用户名\Documents\ai-university-knowledge-vault-test
```

路径不要猜，直接在文件管理器里复制。

### 第三步：选择连接方式

不同工具的配置入口不同。你不用记所有细节，只需要知道自己要填三类信息：

- 资料包路径：也就是 Obsidian Vault 所在文件夹。
- 运行方式：由你使用的 MCP 插件或服务提供。
- 权限范围：建议先只读。

如果你使用 Codex，可以参考 [[Codex配置示例]]。如果你使用其他客户端，就按它自己的 MCP 文档填写。

### 第四步：先做只读测试

连接完成后，先问 5 个不会改文件的问题：

```text
请读取我的测试知识库，只做只读检查：
1. 列出根目录有哪些主要板块。
2. 找到“大学全景图”并概括它连接了哪些核心节点。
3. 找出 5 篇含有 状态:: 规划中 的笔记。
4. 检查是否存在缺少 下一步行动:: 的文件。
5. 不要修改任何文件。
```

如果这些问题能正常回答，说明读取基本成功。

### 第五步：再尝试小范围写入

只读没问题后，再选 1-2 个测试文件做写入：

```text
请只修改测试库中的这 1 篇文件：复盘周记.md。
任务：
1. 补充缺失的 字段:: 值。
2. 推荐 3 个相关 [[双向链接]]。
3. 在文末增加“下周最小行动”。
不要修改其他文件。修改前先告诉我你准备改哪里。
```

确认结果正常，再考虑扩大范围。

## 可以交给工具做的任务

### 任务 1：检查字段是否统一

```text
请扫描我的知识库，只列出字段名问题，不改文件。
重点检查：
1. 同义字段是否重复，比如“阶段”和“年级阶段”。
2. 哪些核心笔记缺少 类型、状态、阶段、关联目标、下一步行动。
3. 给出字段统一建议。
```

### 任务 2：补双向链接

```text
请检查这些笔记之间是否缺少明显的 [[双向链接]]。
要求：
1. 先列出建议，不直接修改。
2. 每个建议说明理由。
3. 不要为了凑数量硬连。
```

### 任务 3：从周记更新长期笔记

```text
请读取本周的复盘周记，判断哪些内容应该更新到长期笔记。
输出：
1. 应该更新的文件名。
2. 每个文件应该补什么事实。
3. 是否需要新建笔记。
4. 先不要写入，等我确认。
```

## 常见问题

### 连接后读不到文件

先检查路径是否指向资料包根目录，而不是某个子文件夹。根目录里应该能看到 `README.md` 和 `🏠 大学全景图.md`。

### 工具想改太多文件

把任务范围缩小到 1-3 个文件，并明确写上“先列计划，不要直接修改”。第一次写入不要做全库批量处理。

### 担心隐私

建立公开库和私人库两套。公开库放模板和脱敏案例，私人库放真实资料。MCP 测试优先连接公开库或测试库。

### 不知道配置文件怎么写

把你使用的客户端名称、资料包路径、插件名称告诉常用智能助手，让它按你的工具生成配置草稿。密钥一律用环境变量，不要直接写进 Markdown 或 GitHub。
