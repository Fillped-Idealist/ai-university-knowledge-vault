# 维护者审查清单

类型:: 项目维护
状态:: 持续使用
阶段:: 发布前/改版前
关联目标:: [[GitHub开源上传指南]]、[[完整操作手册]]
下一步行动:: 每次发布前按清单检查一次

这篇写给项目维护者。它的作用是防止资料包越改越复杂、越改越像工具说明，而忘了大学生真实使用场景。

## 一、用户视角审查

每次新增或修改内容，先问：

- 一个零基础学生能不能看懂？
- 他知道下一步具体做什么吗？
- 是否有真实大学场景，而不是只有概念？
- 是否避免了一开始就要求配置插件或 MCP？
- 是否有隐私提醒？

## 二、单篇笔记审查

每篇笔记最好包含：

- `类型::`：这是什么笔记。
- `状态::`：规划中、进行中、已完成、待复盘。
- `阶段::`：适合大一、大二、大三、大四，还是全阶段。
- `关联目标::`：它服务哪个目标。
- `下一步行动::`：用户看完能做什么。
- 真实场景：什么时候会用到它。
- 信息清单：应该收集什么。
- 常见坑：容易犯什么错。
- 整理指令：如何把原始材料整理进去。

## 三、手册审查

`README.md` 和 `使用手册.md` 必须回答：

- 这个项目解决什么问题？
- 如何下载安装？
- 如何用 Obsidian 打开？
- 第一次使用只做什么？
- 如何让智能助手带着部署？
- 5 天小项目怎么完成？
- 插件和 MCP 什么时候再学？
- 什么内容不能公开？

## 四、去“工具味”审查

如果一段内容出现这些问题，就要重写：

- 只讲功能，不讲场景。
- 只说“用工具整理”，不说输入什么、产出什么、如何判断好坏。
- 只给大词，比如“提升效率”“沉淀资产”，没有具体例子。
- 让用户一开始就配置复杂工具。
- 没有说明隐私边界。

## 五、开源发布审查

发布前检查：

- 没有真实个人信息。
- 没有 token、API Key、`.env`、证书文件。
- `README.md` 是最新入口。
- `使用手册.md` 足够详细。
- zip 包不包含 `.git`。
- 重要新增文件已从 [[🏠 大学全景图]] 或相关索引链接到。

## 六、推荐迭代方向

优先补这些内容：

- 不同专业真实案例。
- 大一到大四的时间线案例。
- 保研、考研、就业、公职的路径对比案例。
- 实习复盘和简历素材转化示例。
- 实验室、科研、竞赛的过程记录示例。

少补这些内容：

- 复杂自动化脚本。
- 与大学生场景无关的工具教程。
- 没有示例的抽象方法论。
